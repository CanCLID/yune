# Phase 01 — UI Design Contract

> Visual and interaction contract for a terminal/CLI frontend surrogate phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> Scope note: this phase is not a graphical product UI. The contract below governs terminal interaction, deterministic transcript output, human-readable rendering, help/error copy, and accessibility of CLI output only.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none |
| Icon library | none |
| Font | Terminal default monospace |

Design system source: repository scan found a Rust-only workspace with no `components.json`, Tailwind, PostCSS, React, Next.js, Vite, CSS, SCSS, or TSX UI files. Existing CLI output is implemented in `crates/yune-cli/src/render.rs`, `crates/yune-cli/src/transcript.rs`, and `crates/yune-cli/src/main.rs`.

Non-goals for this phase:
- Do not add GUI, native frontend, web frontend, mobile UI, shadcn, Tailwind, CSS, icons, theme tokens, or graphical layout scope.
- Do not use color, cursor movement, animation, terminal control sequences, progress spinners, or TUI widgets.
- Do not treat this CLI surrogate as proof that native frontends such as Squirrel, Weasel, ibus-rime, fcitx-rime, or fcitx5-rime are complete.

## Spacing Scale

Declared values (terminal equivalents; all pixel values remain multiples of 4 for checker compatibility):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px / 0 blank lines | Inline separators inside one logical line; use `: ` after labels and `, ` only inside JSON arrays. |
| sm | 8px / 0 blank lines | Compact grouping within help examples and one-event transcript fields. |
| md | 16px / 1 blank line | Separate usage blocks, command groups, or a human-rendered event block from the next block. |
| lg | 24px / 1 blank line | Separate major human-readable transcript sections: setup, event stream, final summary. |
| xl | 32px / 2 blank lines | Reserved for help output between synopsis and detailed examples if detail is added. |
| 2xl | 48px / not used | Not used in this phase; terminal output must stay compact. |
| 3xl | 64px / not used | Not used in this phase; terminal output must stay compact. |

Exceptions:
- Deterministic JSON output must use exactly two spaces per nesting level and no alignment padding beyond JSON indentation.
- Machine-readable JSON stdout must end with exactly one trailing newline.
- Human-readable output must not depend on terminal width; wrap only at natural line boundaries and keep individual lines at or below 100 columns where feasible.

## Typography

Terminal typography is constrained to semantic text roles rather than custom fonts. Use exactly these four roles in CLI rendering and documentation.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | terminal default / 16px equivalent | regular / 400 equivalent | 1.5 equivalent; one logical item per line |
| Label | terminal default / 16px equivalent | regular / 400 equivalent | 1.5 equivalent; lowercase snake_case labels matching JSON keys |
| Heading | terminal default / 16px equivalent | semibold / 600 equivalent by plain text position only | 1.2 equivalent; short section title followed by details |
| Display | terminal default / 16px equivalent | semibold / 600 equivalent by command name position only | 1.2 equivalent; command synopsis only |

Typography contract:
- Use plain ASCII punctuation for structural labels in human output: `usage:`, `commands:`, `error:`, `next:`.
- Do not use bold, italic, underlines, ANSI color, Unicode box drawing, or emoji.
- Preserve RIME/user text as UTF-8 data values; do not transliterate candidate text, preedit text, commits, schema names, or comments.
- For accessibility and copy/paste reliability, every meaningful state must be visible in text without relying on color, cursor position, or glyph styling.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | terminal default background | All CLI output surfaces; no background control. |
| Secondary (30%) | terminal default foreground | Body text, labels, command names, transcript fields, JSON keys and values. |
| Accent (10%) | not used | Reserved text markers only; no color. |
| Destructive | not used | No destructive actions in this phase. |

Accent reserved for: none. If a future CLI mode needs emphasis, use explicit words such as `selected: true` or `highlighted: <index>` instead of color. Candidate highlight must be represented by the deterministic `highlighted` index and, in human output, by a textual marker such as `selected: yes`; never by color alone.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Run transcript |
| Empty state heading | No key events supplied |
| Empty state body | Provide a key sequence or transcript file, then run the command again. |
| Error state | error: {problem}. next: {specific corrective action}. |
| Destructive confirmation | none: this phase has no destructive CLI actions. |

Copy tone:
- Use concise developer-facing language.
- Prefer concrete nouns from the ABI and transcript domain: `shared_data_dir`, `user_data_dir`, `schema_id`, `session_id`, `key`, `commit`, `preedit`, `candidates`, `highlighted`, `status`.
- Error messages must identify the failing operation and the next action. Examples: `error: missing shared data directory. next: pass --shared-data-dir <path>.` and `error: unknown command: replay. next: run yune-cli --help.`
- Help text must start with `usage:` and list commands in stable order.
- Do not anthropomorphize the tool or use celebratory copy.

Command/help contract:
- `--help`, `-h`, and `help` print help to stdout and exit successfully.
- Unknown commands print the error plus help text to stderr and exit with failure.
- Runtime/setup/replay failures print exactly one problem block to stderr and exit with failure.
- Successful machine-readable commands write only JSON to stdout and no stderr.
- Successful human-readable commands write only human transcript text to stdout and no stderr.

## CLI Output And Transcript Contract

### Output modes

| Mode | stdout | stderr | Stability |
|------|--------|--------|-----------|
| Machine JSON | Deterministic JSON object only | Empty on success | Golden-fixture stable; safe for diffing and scripts. |
| Human transcript | Plain text event transcript only | Empty on success | Stable field order; readable in terminals and CI logs. |
| Help | Plain text usage/help only | Empty | Stable command order. |
| Failure | Empty unless failure happens after explicit partial-output mode is introduced | Plain text `error:` block | Stable prefix and next-step wording. |

Separation rules:
- Do not mix human explanatory prose into JSON mode.
- Do not emit warnings, diagnostics, progress lines, or timing information to stdout in JSON mode.
- If diagnostics are needed later, they must go to stderr and must not appear on successful deterministic fixture paths.
- JSON key order is part of the contract and must remain stable for fixture comparison.

### JSON transcript schema

Top-level JSON output for deterministic transcript comparison must include, in this order:
1. `schema_id`
2. `sequence` or transcript source identifier
3. `events`
4. `commits`
5. `context`
6. `status`

`events` records per-key ABI-visible state for replay comparison. `commits`, `context`, and `status` summarize the final state after the event stream so scripts can compare both transition-level and final state data.

`context` must include, in this order:
1. `input`
2. `caret`
3. `preedit`
4. `highlighted`
5. `last_commit`
6. `candidates`

Each candidate must include, in this order:
1. `text`
2. `comment`
3. `source`
4. `quality`

`status` must include, in this order:
1. `schema_id`
2. `schema_name`
3. `is_disabled`
4. `is_composing`
5. `is_ascii_mode`
6. `is_full_shape`
7. `is_simplified`
8. `is_traditional`
9. `is_ascii_punct`

JSON formatting rules:
- Use two-space indentation.
- Escape strings as JSON, including quotes, backslashes, newlines, tabs, and control characters.
- Use `null` for absent optional strings.
- Use `[]` for empty arrays.
- Do not include timestamps, elapsed durations, memory addresses, randomized IDs, absolute temp paths, or environment-dependent values in deterministic output.

### Human transcript rendering

Human-readable rendering must be line-oriented and diffable. For each processed key event, render fields in this order:
1. event index and key label
2. handled/unhandled result if available
3. commit text if present
4. preedit and caret
5. highlighted candidate index
6. candidate page entries
7. status summary

Candidate page rendering:
- Use zero-based indices if matching internal/JSON `highlighted`; if a one-based display is chosen later, label it explicitly and keep JSON zero-based.
- Include candidate `text`, `comment`, `source`, and `quality` where available.
- Mark selected/highlighted candidate with text, not color. Preferred marker: `selected: yes`.
- If no candidates exist, render `candidates: none` rather than omitting the section.

## Interaction States

| State | Required behavior |
|-------|-------------------|
| Initial setup | CLI accepts explicit shared and user data paths for RIME service setup; missing required paths produce an `error:` plus `next:` message. |
| Deploy/select schema | Deployment and schema selection outcomes are visible through success output or actionable failure text; schema identifiers remain exact and case-preserving. |
| Session lifecycle | Session creation, key processing, and destruction happen through `yune-rime-api`; user-visible output reports resulting state, not implementation internals. |
| Key processed with commit | Render commit text and resulting cleared or updated composition state. |
| Key processed while composing | Render preedit, caret, candidate page, highlighted index, and status after the event. |
| Key ignored/unhandled | Render or encode the unhandled result without pretending state changed. |
| Transcript replay match | Output deterministic JSON suitable for fixture comparison and exit successfully. |
| Transcript replay mismatch | Exit with failure and identify the first differing field or fixture path; do not dump unrelated debug state. |
| No data | Use the empty state copy defined above. |

---

## Accessibility Contract

- Output must be useful in screen readers, CI logs, pipes, and plain text files.
- Do not rely on color, icons, cursor movement, spatial tables, or live-updating terminal regions to communicate status.
- Keep each logical field on its own line in human transcript mode.
- Use stable labels before values so users can search logs with plain text tools.
- Do not truncate candidate text, commit text, preedit text, schema names, or error paths in deterministic modes.
- Avoid ambiguous symbols. Prefer `true`/`false`, `none`, `null`, and explicit indices over checkmarks, crosses, arrows, or colored badges.
- Preserve Unicode input-method data while keeping structural output ASCII.
