<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Runtime/session commit to persistent userdb | User text and selected candidates cross from transient composition into persistent user dictionary state. |
| Core engine to `yune-rime-api` storage | Storage-agnostic learning metadata crosses into runtime path/userdb persistence owned by the API crate. |
| ABI frontend flow to candidate learning | Untrusted frontend key/commit sequence can trigger learning and later candidate ordering changes. |
| Userdb lookup to candidate list | Persistent learned entries influence visible candidates and quality ordering. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-02-01 | Tampering | Session commit to userdb store | mitigate | Task 2 validates schema/user dictionary logical IDs before selecting storage and persists only typed commit metadata through Plan 05-01 store APIs. |
| T-05-02-02 | Repudiation | Learning update provenance | mitigate | Task 1 records candidate type/source, input, selected text, and tick-like metadata before clear; tests prove commit-driven updates are attributable to normal session flow. |
| T-05-02-03 | Denial of Service | Predictive/backdated lookup | mitigate | Task 1 defines explicit backdated scan scope and bounded predictive lookup from current engine context; tests ensure unrelated histories/AI memory are not scanned. |
| T-05-02-04 | Tampering/Elevation | Optional rankers and AI substitutes | mitigate | Tasks 1-3 grep/test gates ensure `HistoryTranslator`, `CandidateRanker`, `MockAiRanker`, and future AI memory APIs are not required substitutes for classic userdb behavior. |
| T-05-02-05 | Information Disclosure | Learned candidate output | accept | Learned entries are local userdb candidates visible through normal IME behavior; this plan adds no network/provider export path. |
| T-05-02-06 | FFI invalid pointer misuse | Frontend client/session ABI | mitigate | Task 2 proves behavior through existing `rime_get_api`/session ABI tests and does not add new unsafe pointer surfaces for learning. |
</threat_model>
