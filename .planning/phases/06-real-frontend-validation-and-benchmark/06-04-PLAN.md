<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Benchmark harness -> RIME ABI function table | Benchmark code crosses C ABI-like function pointer and allocation ownership boundaries through `RimeApi`. |
| Local synthetic fixtures -> repository docs | Benchmark and readiness artifacts convert local runtime observations into committed documentation. |
| Userdb fixture setup -> local filesystem | Userdb learning/sync benchmarks create and mutate local fixture files under temp or repository-controlled fixture directories. |
| Phase 6 evidence -> AI-native readiness recommendation | Documentation converts validation and benchmark evidence into a product-readiness decision without designing AI implementation. |
| Local benchmark execution -> external services | Benchmark runs must remain local and must not depend on remote/cloud services, desktop daemons, browser workers, or external frontend installations. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-04-01 | Information Disclosure | `docs/frontend-validation/benchmark-baselines.md` and benchmark output | mitigate | Sanitize benchmark artifacts: omit absolute temp/home paths, secrets, raw pointers, process IDs, noisy timestamps, and real user dictionary contents; use synthetic fixture names and controlled record counts only. |
| T-06-04-02 | Tampering | `crates/yune-rime-api/benches/frontend_baselines.rs` userdb fixture setup | mitigate | Create userdb learning/sync fixtures from deterministic local synthetic records; validate logical userdb/schema IDs; do not read or mutate real user data directories. |
| T-06-04-03 | Denial of Service | Benchmark loops and deployment/dictionary/userdb scenarios | mitigate | Use bounded operation counts and fixture sizes documented in the baseline; avoid unbounded loops, network waits, desktop daemon waits, or external frontend process dependencies. |
| T-06-04-04 | Information Disclosure | Local userdb fixture data | mitigate | Populate fixture data with synthetic phrases only; never copy real userdb files into docs or benchmark outputs; ensure docs contain counts and scenario names rather than raw personal dictionaries. |
| T-06-04-05 | Spoofing | Readiness recommendation evidence | mitigate | Base recommendation on actual 06-01/06-02/06-03 summaries and Task 2 benchmark results; link evidence files and record blockers instead of presenting unavailable validation as success. |
| T-06-04-06 | Elevation of Privilege | Remote/cloud dependencies or external frontend tooling | mitigate | Keep BENCH-01/BENCH-02 harness local and dependency-minimal; do not add network calls, cloud inference, browser worker requirements, Squirrel, ibus, fcitx, or daemon setup to automated benchmark execution. |
| T-06-04-07 | Repudiation | Benchmark comparability | mitigate | Record commands, toolchain/platform/profile metadata, operation counts, fixture names, and blocker status so future changes can compare against or reproduce the baseline. |
| T-06-04-08 | Tampering | `crates/yune-rime-api/Cargo.toml` benchmark configuration | mitigate | Preserve `crate-type = ["rlib", "cdylib"]`; only add MSRV-safe benchmark configuration/dependencies after verifying Rust 1.76 compatibility; prefer dependency-free harness if compatibility is uncertain. |
</threat_model>
