# Plan 08-02: Add Deterministic TypeDuck Runtime Tests

## Goal

Add package-local tests for the TypeScript TypeDuck runtime package created in 08-01. The tests must prove wrapper behavior, response ownership, lifecycle guards, and browser-key mapping without requiring Emscripten, a browser, TypeDuck-Web source, schema assets, or real WASM output.

## Context

Phase 8 owns the TypeScript bridge contract around the Phase 7 `yune_typeduck_*` adapter symbols. Tests should exercise the wrapper with a fake injected Module because Phase 7 already verifies exported Rust symbols and Phase 10 owns real browser integration.

Canonical inputs:

- `.planning/phases/08-typescript-bridge-and-runtime-package/08-CONTEXT.md`
- `.planning/phases/08-typescript-bridge-and-runtime-package/08-RESEARCH.md`
- `.planning/phases/08-typescript-bridge-and-runtime-package/08-PATTERNS.md`
- `.planning/phases/08-typescript-bridge-and-runtime-package/08-01-PLAN.md`
- `docs/typeduck-web-adapter.md`
- `scripts/typeduck-exports.txt`
- `crates/yune-rime-api/tests/typeduck_web.rs`

## Scope

In scope:

- Fake Emscripten Module test harness.
- Tests for `bindTypeDuckModule` and missing exports.
- Tests for centralized response parsing/freeing.
- Tests for malformed/null response paths.
- Tests for runtime init, operation forwarding, candidate actions, deploy/customize, cleanup, and post-cleanup guards.
- Tests for deterministic DOM-free keyboard event mapping.

Out of scope:

- Real Emscripten builds.
- Browser or DOM test environment.
- TypeDuck-Web checkout or app patching.
- MEMFS/IDBFS setup, asset preload, or persistence sync.
- Rust adapter changes.
- Generated bindings.

## Must Haves

- Tests must run with `npm --prefix packages/yune-typeduck-runtime test`.
- Tests must not require DOM globals, real `KeyboardEvent`, browser APIs, Emscripten, WASM, or TypeDuck-Web.
- The fake module must expose the same narrow interface as production code: `cwrap` and `UTF8ToString`.
- Response-producing operations must prove `yune_typeduck_free_response` is called exactly once for each nonzero response pointer.
- Response tests must prove freeing still happens when JSON parsing or validation fails.
- Null response pointers must throw deterministic wrapper errors and must not call free for pointer `0`.
- Null JSON pointers must throw deterministic wrapper errors and must free the owning response pointer.
- Runtime tests must prove `cleanup()` is idempotent at the TypeScript layer so the same raw state pointer is not passed twice to Rust cleanup.
- Runtime tests must prove non-cleanup operations after cleanup throw `TypeDuck runtime has been cleaned up`.
- Key mapping tests must assert use of `event.key` semantics and the source package must still contain no `keyCode` references.

## Tasks

### 1. Add the fake Emscripten Module test harness

Create `packages/yune-typeduck-runtime/test/fake-module.ts`.

Implement a fake module that satisfies `EmscriptenTypeDuckModule` from `src/module.ts`.

The fake should support:

- Registering wrapped exports by exact symbol name.
- `cwrap(ident, returnType, argTypes)` returning the registered function.
- Throwing when a test asks for a missing fake export unexpectedly.
- `UTF8ToString(ptr)` returning string memory for a numeric pointer.
- Allocating fake string pointers for JSON payloads.
- Allocating fake response pointers that map to:
  - JSON pointer
  - handled flag
  - freed state
- Recording every call to:
  - `yune_typeduck_init`
  - `yune_typeduck_process_key`
  - `yune_typeduck_select_candidate`
  - `yune_typeduck_delete_candidate`
  - `yune_typeduck_flip_page`
  - `yune_typeduck_deploy`
  - `yune_typeduck_customize`
  - `yune_typeduck_cleanup`
  - `yune_typeduck_free_response`

Suggested test helper shape:

```typescript
export class FakeTypeDuckModule implements EmscriptenTypeDuckModule {
  cwrap(ident: string): EmscriptenWrappedFunction;
  UTF8ToString(ptr: number): string;
  response(json: unknown, handled?: boolean): number;
  responseWithJsonPointer(jsonPtr: number, handled?: boolean): number;
  string(value: string): number;
  freedResponses(): number[];
  calls(symbol: string): unknown[][];
}
```

Keep the fake small and deterministic. Do not simulate RIME behavior beyond returning configured pointers and recording calls.

### 2. Test module binding behavior

Add binding tests in `packages/yune-typeduck-runtime/test/typeduck.test.ts` or a separate section of that file.

Cover:

- `bindTypeDuckModule` binds all 11 symbols from `scripts/typeduck-exports.txt`.
- The bound functions call the exact `yune_typeduck_*` fake exports.
- Missing exports surface as `TypeDuckBindingError` with message `Missing TypeDuck export: <symbol>`.
- Bound boolean-ish C returns are exposed as numeric `0`/nonzero values for runtime code to interpret.

The fake module may expose a helper to delete one registered export for the missing-export case.

### 3. Test centralized response parsing and ownership

Create `packages/yune-typeduck-runtime/test/response.test.ts`.

Cover `readTypeDuckResponse` directly with fake bindings.

Required cases:

1. Parses a valid adapter response object:
   - `handled: true`
   - `commits: ["你"]`
   - nullable or populated `context`
   - nullable or populated `status`
   - optional `error`
2. Uses `yune_typeduck_response_handled` as the authoritative handled value if JSON and native handled disagree.
3. Calls `yune_typeduck_free_response(responsePtr)` exactly once after a successful parse.
4. Throws `TypeDuckResponseError("TypeDuck adapter returned null response")` for response pointer `0` and does not free pointer `0`.
5. Throws `TypeDuckResponseError("TypeDuck adapter returned null response JSON")` when `responseJson(responsePtr)` returns `0`, and still frees `responsePtr` exactly once.
6. Throws a deterministic `TypeDuckResponseError` for malformed JSON text and still frees `responsePtr` exactly once.
7. Throws a deterministic `TypeDuckResponseError` for non-object JSON and still frees `responsePtr` exactly once.
8. Throws a deterministic `TypeDuckResponseError` when `handled` is not boolean or `commits` is not an array, and still frees `responsePtr` exactly once.

Keep validation focused on the documented response envelope. Do not over-validate every nested candidate/status field in Phase 8.

### 4. Test runtime init and operation forwarding

Extend `packages/yune-typeduck-runtime/test/typeduck.test.ts`.

Cover:

- `TypeDuckRuntime.init(module, { sharedDataDir, userDataDir, schemaId })` calls `yune_typeduck_init` with those strings.
- Init pointer `0` throws `TypeDuckLifecycleError("TypeDuck adapter init failed")` and does not expose a runtime object.
- `processKey(keycode, mask)` forwards `statePtr`, `keycode`, and `mask` to `yune_typeduck_process_key`.
- `processKey(keycode)` defaults mask to `0`.
- `processKeyboardEvent(event)` maps through `keyEventToRimeKey` and then calls `processKey`.
- `selectCandidate(index)` forwards a page-relative index to `yune_typeduck_select_candidate`.
- `deleteCandidate(index)` forwards a page-relative index to `yune_typeduck_delete_candidate`.
- `flipPage(false)` passes `0` to `yune_typeduck_flip_page`.
- `flipPage(true)` passes `1` to `yune_typeduck_flip_page`.
- `deploy()` returns `true` for nonzero adapter return and `false` for `0`.
- `customize(configId, key, value)` forwards strings and returns boolean from nonzero/zero adapter return.

For response-producing runtime methods, assert the returned parsed response matches fake JSON and the response pointer is freed exactly once.

### 5. Test runtime lifecycle guards

In `packages/yune-typeduck-runtime/test/typeduck.test.ts`, add lifecycle-focused cases.

Cover:

- `cleanup()` calls `yune_typeduck_cleanup(statePtr)` once.
- Calling `cleanup()` twice only calls the fake cleanup export once.
- After cleanup, all non-cleanup methods throw an error containing `TypeDuck runtime has been cleaned up`:
  - `processKey`
  - `processKeyboardEvent`
  - `selectCandidate`
  - `deleteCandidate`
  - `flipPage`
  - `deploy`
  - `customize`
- Cleanup after a failed init is impossible because no runtime object is returned.

This enforces the TypeScript-side guard around Rust's raw cleanup ownership, where the adapter consumes the state pointer.

### 6. Test key mapping deterministically

Create `packages/yune-typeduck-runtime/test/keys.test.ts`.

Cover `keyEventToRimeKey` and exported constants from `src/keys.ts`.

Required mapping tests:

- Printable lowercase key:
  - `{ key: "a" }` -> keycode `"a".charCodeAt(0)`, mask `0`
- Printable uppercase key with shift:
  - `{ key: "A", shiftKey: true }` -> keycode `"A".charCodeAt(0)`, mask includes Shift
- Digit selection key:
  - `{ key: "1" }` -> keycode `49`, mask `0`
- Space:
  - `{ key: " " }` and `{ key: "Space" }` -> `0x20`
- Editing/navigation keys:
  - `Backspace` -> `0xff08`
  - `Tab` -> `0xff09`
  - `Enter` -> `0xff0d`
  - `Escape` -> `0xff1b`
  - `Delete` -> `0xffff`
  - `ArrowLeft` -> `0xff51`
  - `ArrowUp` -> `0xff52`
  - `ArrowRight` -> `0xff53`
  - `ArrowDown` -> `0xff54`
  - `PageUp` -> `0xff55`
  - `PageDown` -> `0xff56`
  - `Home` -> `0xff50`
  - `End` -> `0xff57`
- Modifiers:
  - Shift sets `1 << 0`
  - Control sets `1 << 2`
  - Alt sets `1 << 3`
  - Meta sets `1 << 28`
  - Release event type sets `1 << 30` if `type` is modeled in 08-01
- Unknown multi-character key throws a deterministic key mapping error instead of guessing.

Do not use real `KeyboardEvent` objects. Use the narrow `TypeDuckKeyboardEventLike` object type.
