# Plan 08-03: Document TypeScript Runtime Lifecycle And Final Gates

## Goal

Update the TypeDuck-Web adapter documentation so browser callers can use the TypeScript runtime package safely, understand the one-active-service lifecycle constraint, and know which host responsibilities remain deferred to later phases. Finish Phase 8 with package-level build/test gates and documentation checks.

## Context

Plans 08-01 and 08-02 create and test a minimal repository-owned TypeScript bridge package around the Phase 7 `yune_typeduck_*` adapter symbols. This plan documents the resulting caller contract and verifies the complete Phase 8 package surface without expanding into Phase 9 filesystem orchestration or Phase 10 TypeDuck-Web app integration.

## Scope

In scope:

- Documenting the package-local TypeScript runtime wrapper.
- Documenting how callers inject the Emscripten Module after it is ready.
- Documenting wrapper operations for init, key processing, candidate actions, deploy, customize, and cleanup.
- Documenting response ownership as handled by the TypeScript wrapper.
- Documenting deterministic key mapping API and its limits.
- Documenting one-active-process-global-service lifecycle constraints.
- Documenting browser filesystem and persistence responsibilities that remain host-owned.
- Running final package build/test and documentation checks.

Out of scope:

- Adding a root README or root JS workspace.
- Adding browser app scaffolding.
- Mounting MEMFS/IDBFS or implementing persistence sync.
- Cloning, vendoring, or patching upstream TypeDuck-Web.
- Browser E2E tests.
- Rust adapter behavior changes.
- AI-native ranking/provider/context/memory/privacy behavior.

## Must Haves

- `docs/typeduck-web-adapter.md` must mention `@yune-ime/typeduck-runtime` and its package-local path.
- Documentation must show a TypeScript wrapper flow that replaces raw manual response-free snippets for wrapper users.
- Documentation must still retain the low-level C/WASM export contract for hosts that do not use the wrapper.
- Documentation must explicitly state that `cleanup()` finalizes the process-global Yune/RIME service and that callers should treat one Module instance as having one active Yune TypeDuck runtime at a time.
- Documentation must state that TypeScript `cleanup()` is idempotent at the wrapper layer, but raw `yune_typeduck_cleanup(statePtr)` is not caller-managed through the wrapper.
- Documentation must state that operations after wrapper cleanup fail deterministically.
- Documentation must state that response pointers remain opaque and wrapper callers do not call `yune_typeduck_free_response` directly.
- Documentation must state that `processKeyboardEvent` accepts a narrow event-like object and uses `event.key` semantics, not deprecated `keyCode`.
- Documentation must preserve Phase 9 deferrals: browser filesystem setup, schema/dictionary asset preload, IDBFS sync, persistence recovery.
- Documentation must preserve Phase 10 deferrals: upstream TypeDuck-Web checkout/patching and real browser E2E.

### 2. Document wrapper initialization and Module injection

Document that callers construct the runtime only after the Emscripten Module is initialized and exposes `cwrap` and `UTF8ToString`.

Include a concise TypeScript example:

```typescript
const runtime = TypeDuckRuntime.init(Module, {
  sharedDataDir: "/yune/shared",
  userDataDir: "/yune/user",
  schemaId: "luna_pinyin",
});
```

State that the wrapper binds only the canonical `yune_typeduck_*` symbols and leaves Emscripten build/export retention to the Phase 7 build script and host build flags.

### 3. Document wrapper operations

Document the public wrapper operations and their adapter mapping:

- `TypeDuckRuntime.init(...)` -> `yune_typeduck_init`
- `runtime.processKey(keycode, mask)` -> `yune_typeduck_process_key`
- `runtime.processKeyboardEvent(event)` -> key mapping + `processKey`
- `runtime.selectCandidate(index)` -> `yune_typeduck_select_candidate`
- `runtime.deleteCandidate(index)` -> `yune_typeduck_delete_candidate`
- `runtime.flipPage(backward)` -> `yune_typeduck_flip_page`
- `runtime.deploy()` -> `yune_typeduck_deploy`
- `runtime.customize(configId, key, value)` -> `yune_typeduck_customize`
- `runtime.cleanup()` -> `yune_typeduck_cleanup`

State candidate indices are page-relative, matching the native adapter contract.

### 4. Document response ownership at the wrapper boundary

Add a wrapper-specific response ownership subsection.

State:

- Low-level C/WASM adapter response pointers are owned and must be freed exactly once.
- Wrapper callers receive parsed `TypeDuckResponse` objects, not raw response pointers.
- The wrapper copies JSON using `UTF8ToString`, parses it, and calls `yune_typeduck_free_response` in a centralized `finally` path.
- Null response pointers, null JSON pointers, malformed JSON, and malformed response envelopes are surfaced as deterministic wrapper errors.
- Callers that bypass the wrapper must still follow the existing low-level rule: copy JSON before `yune_typeduck_free_response`.

Do not imply the wrapper hides missing assets or runtime errors. Missing assets should remain visible until Phase 9 recovery work exists.

### 5. Document keyboard mapping

Add a section for browser-key mapping.

State:

- The wrapper exposes `keyEventToRimeKey(event)` and `processKeyboardEvent(event)`.
- The accepted event is a narrow DOM-free object with `key`, optional `shiftKey`, `ctrlKey`, `altKey`, `metaKey`, and optional `type`.
- Mapping uses `event.key`, not deprecated `keyCode`.
- Phase 8 covers printable keys, Space, Enter, Backspace, Escape, Delete, arrows, PageUp/PageDown, Home/End, digit selection keys, and common modifiers.
- Exhaustive cross-browser keyboard edge cases are deferred until Phase 10 observes the real TypeDuck-Web seam.

Include a short example:

```typescript
const response = runtime.processKeyboardEvent({
  key: event.key,
  shiftKey: event.shiftKey,
  ctrlKey: event.ctrlKey,
  altKey: event.altKey,
  metaKey: event.metaKey,
  type: event.type,
});
```

### 6. Document lifecycle constraints prominently

Update or add a lifecycle section that states:

- Yune/RIME service state is process-global in this adapter milestone.
- A page/worker should treat one Emscripten Module instance as having one active `TypeDuckRuntime` at a time.
- `runtime.cleanup()` finalizes the process-global service through `yune_typeduck_cleanup`.
- `runtime.cleanup()` is idempotent at the TypeScript layer.
- After cleanup, wrapper operations throw deterministic lifecycle errors instead of calling the consumed native state pointer.
- Multiple simultaneous runtimes with different data directories are out of scope and may interfere.

This must align with Phase 7 D-10 and Phase 8 D-13/D-14.

### 7. Preserve browser host responsibility boundaries

Ensure `docs/typeduck-web-adapter.md` still states browser hosts are responsible for:

- Creating `sharedDataDir`.
- Creating `userDataDir`.
- Creating `userDataDir/build`.
- Preloading schema and dictionary assets before init.
- Syncing IDBFS or other persistence before init and after deploy/customize/userdb changes.
- Deciding worker/page lifetime and cleanup timing.

State that Phase 9 owns MEMFS/IDBFS setup, asset preload, persistence sync, and recovery paths.

### 8. Preserve downstream deferrals

- Real browser E2E to Phase 10.

Do not add implementation promises for these deferred areas.
