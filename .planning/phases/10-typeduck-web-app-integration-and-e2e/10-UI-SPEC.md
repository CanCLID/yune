# Phase 10 — UI Design Contract

> E2E/design contract for preserving and validating the upstream TypeDuck-Web browser UI while routing its existing input-engine seam to Yune. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> Scope note: this phase is not a Yune UI redesign. The contract below governs what the real upstream TypeDuck-Web app must continue to render and how browser E2E must observe composition, candidate paging, candidate selection, candidate deletion, commit output, deploy, customize, and persistence smoke behavior through the Yune bridge.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | upstream TypeDuck-Web UI under test; no Yune-owned design system |
| Preset | not applicable |
| Component library | preserve upstream TypeDuck-Web React components and styles |
| Icon library | preserve upstream TypeDuck-Web icons if present; add none for Yune |
| Font | preserve upstream TypeDuck-Web font stack |

Design system source: repository scan of the Yune worktree found no `components.json`, Tailwind config, PostCSS config, CSS/SCSS files, TSX UI files, React app, Next.js app, Vite app, or shadcn setup. Phase 10 integrates the existing upstream TypeDuck-Web app as the frontend under test. Therefore the visual contract is preservation: do not introduce shadcn, Tailwind tokens, first-party Yune components, new visual styling, or a replacement layout unless a minimal upstream patch is required for stable E2E selectors or bridge payload wiring.

Upstream/app behavior sources used for this contract:

- `.planning/phases/10-typeduck-web-app-integration-and-e2e/10-CONTEXT.md`: Phase 10 is a real TypeDuck-Web integration and E2E phase, not a new Yune frontend.
- `.planning/phases/10-typeduck-web-app-integration-and-e2e/10-RESEARCH.md`: TypeDuck-Web seam is `src/rime.ts`, `src/worker.ts`, `src/types.ts`, `src/CandidatePanel.tsx`, `src/App.tsx`, and the upstream Emscripten bridge.
- `.planning/ROADMAP.md` and `.planning/REQUIREMENTS.md`: Phase 10 must validate composition, candidate paging, selection, deletion, commit output, deploy, customize, and persistence smoke.
- `docs/typeduck-web-adapter.md`: Yune exposes the `@yune-ime/typeduck-runtime` wrapper and explicit filesystem/persistence helpers consumed by TypeDuck-Web glue.

Non-goals for this phase:

- Do not design or build a new first-party Yune graphical frontend.
- Do not redesign TypeDuck-Web visual layout, color palette, typography, spacing, toolbar, preferences, candidate panel, or app navigation.
- Do not add AI-native provider calls, AI candidates, AI ranking, context capture, memory surfaces, privacy controls, source labels, model settings, or cloud/local model UI.
- Do not introduce Yune-specific product policy for CDN/cache/service-worker/storage-quota behavior beyond the minimum needed to run TypeDuck-Web E2E.
- Do not claim native fallback tests, package-local fake-module tests, or static seam inspection satisfy the real browser E2E contract.

## Spacing Scale

Declared values (checker-compatible multiples of 4; preservation contract for upstream UI rather than new tokens):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Minimum inline gaps if adding `data-testid` wrappers or nonvisual bridge-only labels; do not change upstream spacing for visual reasons. |
| sm | 8px | Compact upstream-equivalent control gaps only when a minimal selector wrapper needs spacing; otherwise preserve app CSS. |
| md | 16px | Default spacing reference for any temporary E2E status panel or blocker banner if upstream already has an equivalent surface. |
| lg | 24px | Section spacing reference for existing TypeDuck-Web preferences/deploy surfaces; do not add new sections. |
| xl | 32px | Existing page-level gap reference only; no new Yune layout blocks. |
| 2xl | 48px | Reserved; do not use unless upstream already uses comparable spacing. |
| 3xl | 64px | Reserved; do not use unless upstream already uses comparable spacing. |

Exceptions:

- Preserve TypeDuck-Web's existing CSS spacing even when it does not map exactly to this scale. The purpose of this phase is upstream UI fidelity, not token normalization.
- Stable test selectors (`data-testid`, `aria-label`, or role/name improvements) must be visually neutral: no padding, margin, size, or layout changes unless the existing UI cannot expose the required browser state otherwise.
- Candidate interaction hit targets must remain at least as usable as upstream. If a new wrapper is unavoidable around candidate rows, page buttons, deploy controls, or preference controls, maintain a minimum 44px pointer target where the upstream control already behaves as a touch/click target.
- E2E assertions must not fail on purely cosmetic spacing drift unless it prevents interaction or hides required state.

## Typography

Typography is a preservation contract. Do not add new Yune fonts or text hierarchy. Browser E2E should rely on accessible roles/names and visible text, not on font metrics.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px equivalent | regular / 400 equivalent | 1.5 |
| Label | 14px equivalent | regular / 400 equivalent | 1.5 |
| Heading | 20px equivalent | semibold / 600 equivalent | 1.2 |
| Display | 28px equivalent | semibold / 600 equivalent | 1.2 |

Typography contract:

- Preserve upstream TypeDuck-Web text rendering for textarea input, preedit/composition text, candidate text, candidate comments, page controls, toolbar labels, preference labels, notification text, and deploy/customize status.
- Preserve UTF-8 composition, candidate, comment, and commit strings exactly. Do not transliterate, normalize, truncate, or replace candidate text for test convenience.
- New E2E-only accessible labels must be concise and behavior-named, for example `composition input`, `candidate 0`, `next candidate page`, `previous candidate page`, `delete candidate 0`, `deploy`, and `preferences`.
- Do not use emoji, decorative glyphs, ASCII art, or stylized Yune branding in TypeDuck-Web integration copy.
- Do not communicate test-critical state by font weight or styling alone. Every required state must be observable through DOM text, accessible name, `data-testid`, or bridge-level state exposed to the browser test.

## Color

Color is a preservation contract. Do not introduce a Yune palette. The 60/30/10 roles below describe how to reason about upstream TypeDuck-Web surfaces if a minimal selector or status copy patch is needed.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | upstream TypeDuck-Web page background | Existing app background and primary editing surface. |
| Secondary (30%) | upstream TypeDuck-Web panels/cards/nav surfaces | Existing candidate panel, toolbar, preferences, notifications, and deploy/customize surfaces. |
| Accent (10%) | upstream TypeDuck-Web existing accent color | Reserved only for the existing focused candidate, active page/control state, selected preference control, and existing deploy/customize status emphasis. |
| Destructive | upstream TypeDuck-Web destructive/warning color if present | Candidate deletion affordance only; no new destructive styling. |

Accent reserved for: focused/highlighted candidate, candidate page active state, selected preference/toggle state, deploy/customize status emphasis already present upstream. Do not use accent for all buttons, all links, or new Yune branding.

Destructive action contract:

- Candidate deletion is the only destructive user-visible flow in Phase 10.
- Preserve upstream deletion interaction model. If upstream uses long-press, alternate click, context action, or a delete affordance, E2E must exercise that same model instead of adding a new confirmation modal.
- Do not add new red/destructive styling unless the upstream app already has a deletion state that the test can observe.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Run TypeDuck-Web E2E |
| Empty state heading | No composition |
| Empty state body | Type into the existing TypeDuck-Web input area to start composition. |
| Error state | TypeDuck-Web integration failed: {problem}. Next: {specific command, dependency, fixture, or bridge action}. |
| Destructive confirmation | Candidate deletion: preserve upstream deletion behavior; do not add a new confirmation prompt in this phase. |

Copy tone:

- Preserve upstream TypeDuck-Web user-facing copy wherever possible.
- New copy is allowed only for integration/blocker reporting, accessible labels needed by E2E, or deterministic test-only status surfaces.
- Error/blocker copy must identify the failure class: `TypeDuck-Web source blocker`, `Yune adapter/runtime mismatch`, `browser tooling blocker`, `WASM artifact blocker`, `persistence sync blocker`, or `E2E assertion failure`.
- Error/blocker copy must include the next action, for example: install Emscripten, install `wasm32-unknown-emscripten`, rerun `./scripts/typeduck-wasm-build.sh`, preload the missing schema asset, fix the worker bridge mapping, or rerun Playwright with trace artifacts.
- Do not add celebratory, AI-branded, marketing, onboarding, telemetry, remote-inference, or model-provider copy.

Command/blocker copy contract:

- Successful browser validation summary: `TypeDuck-Web E2E passed with Yune bridge: composition, paging, selection, deletion, commit, deploy, customize, persistence smoke.`
- Blocked browser validation summary: `TypeDuck-Web E2E blocked: {dependency_or_artifact}. Fallback: {fallback_command_and_result}. Next: {unblocking_action}.`
- Bridge mismatch summary: `Yune bridge mismatch: {upstream_action} expected {expected_shape}; Yune returned {actual_shape}. Next: {adapter_mapping_or_focused_export_decision}.`
- Persistence failure summary: `TypeDuck-Web persistence failed during {fromPersistence|toPersistence}: {problem}. Next: retry sync or reinitialize only after required assets and persisted state are ready.`

## TypeDuck-Web E2E Interaction Contract

### Surfaces under test

| Surface | Required preservation | E2E observation |
|---------|-----------------------|-----------------|
| Main input/composition area | Preserve upstream text input and composition entry flow. | Browser can focus the input, send key events, and observe composition/preedit state after typing. |
| Candidate panel | Preserve upstream panel visibility, candidate order, highlighted candidate, comments, and page controls. | Browser can locate candidate panel, candidate rows, highlighted row/index, comments where rendered, current page, and previous/next controls. |
| Candidate selection | Preserve upstream click/key selection behavior and page-relative indices. | Browser can select a visible candidate and observe commit output plus cleared or updated composition state. |
| Candidate deletion | Preserve upstream deletion behavior and target candidate semantics. | Browser can delete a visible candidate and observe that candidate disappear, move, or produce the upstream deletion state without adding new confirmation UI. |
| Commit output | Preserve upstream committed text path into the editor/textarea/output area. | Browser can assert committed text appears in the app-visible output and `TypeDuckResponse.commits` was translated correctly if bridge state is exposed. |
| Candidate paging | Preserve upstream previous/next page behavior and disabled/last-page behavior. | Browser can move to another page, observe candidate list/page state changes, and handle last-page disabled state. |
| Deploy | Preserve upstream deploy trigger/status model. | Browser can trigger deploy through existing app UI or app action and observe success, visible status, or deterministic blocker. |
| Customize/preferences | Preserve upstream preference/customize flow. | Browser can change a preference, observe customize/deploy flow, and verify the app remains usable. |
| Persistence smoke | Preserve local-first browser persistence through Emscripten FS/IDBFS or equivalent. | Browser can reload or reinitialize after sync and observe at least one persisted deploy/customize/userdb-relevant state survives. |

### Required browser flows

1. **Initialization and source seam**
   - Given the pinned TypeDuck-Web checkout is installed and served at its existing `/web/` base path,
   - when the app loads with the Yune worker/bridge patch,
   - then the browser observes the existing TypeDuck-Web UI, the worker initializes one Yune `TypeDuckRuntime`, required schema assets are present, and no new Yune product UI is displayed.

2. **Composition**
   - Given the TypeDuck-Web input area is focused,
   - when Playwright sends printable key events such as a deterministic fixture sequence,
   - then the app shows upstream composition/preedit UI, Yune `context.input`/`context.preedit` semantics are reflected in the UI, candidate panel appears when candidates exist, and the input method does not auto-commit unless the schema behavior requires it.

3. **Candidate paging**
   - Given a composition has more candidates than one page,
   - when the browser activates the existing next-page control or sends the upstream page key,
   - then the visible candidate set or page indicator changes, `page_no` translation remains page-relative, and previous/next disabled states match `is_last_page` where observable.

4. **Candidate selection**
   - Given candidates are visible,
   - when the browser selects candidate index `0` through the upstream UI or select key,
   - then the selected candidate commits to the app output, the candidate index is page-relative, `commits` are appended exactly once, and response ownership remains inside `TypeDuckRuntime`.

5. **Candidate deletion**
   - Given candidates are visible and the upstream deletion affordance is available,
   - when the browser performs the upstream delete action for candidate index `0`,
   - then the app updates candidate state without adding a new confirmation dialog, and E2E records the resulting candidate disappearance/reordering/blocker exactly.

6. **Commit output**
   - Given a composition can be committed by Enter, Space, select key, or candidate click according to upstream behavior,
   - when the browser commits,
   - then the committed UTF-8 text appears in the same TypeDuck-Web output/input location used by the upstream app, composition clears or advances as upstream expects, and no duplicate commit is rendered after reload or action retry.

7. **Deploy**
   - Given the app exposes its existing deploy path,
   - when the browser triggers deploy,
   - then worker glue calls `runtime.deploy()` or `deployAndSync(runtime, FS)`, the app shows the existing deploy status/success/failure state, and any failure reports the exact missing tool/artifact/asset or adapter mismatch.

8. **Customize/preferences**
   - Given the app exposes preferences/customization,
   - when the browser changes one supported preference,
   - then worker glue calls `runtime.customize(configId, key, value)` or `customizeAndSync`, the UI reflects the existing preference state, and unsupported upstream options are either mapped intentionally or recorded as focused adapter blockers.

9. **Persistence smoke**
   - Given a deploy/customize or userdb-relevant commit boundary has occurred,
   - when the host calls `syncToPersistenceAfterMutation` and the browser reloads/reinitializes,
   - then initialization calls `syncFromPersistenceBeforeInit`, required assets remain ready, and at least one persisted app-visible state survives the reload.

### Bridge mapping assertions

| Upstream action | Yune operation | Required assertion |
|----------------|----------------|--------------------|
| `processKey` / keyboard event | `runtime.processKeyboardEvent(event)` or bounded compatibility parser to `runtime.processKey(keycode, mask)` | Printable keys, Backspace, Enter, Space, Escape, Delete, arrows, PageUp/PageDown, and digit select keys map deterministically. |
| `selectCandidate(index)` | `runtime.selectCandidate(index)` | Index is page-relative and commit output is rendered once. |
| `deleteCandidate(index)` | `runtime.deleteCandidate(index)` | Index is page-relative and candidate deletion state is visible or blocker is recorded. |
| `flipPage(backward)` | `runtime.flipPage(backward)` | `backward=false` moves next; `backward=true` moves previous; page state is reflected. |
| `customize(preferences)` | `runtime.customize(configId, key, value)` / `customizeAndSync` | Supported preference changes persist or unsupported options are explicit blockers. |
| `deploy()` | `runtime.deploy()` / `deployAndSync` | Deploy status is visible and persistence sync follows successful mutation. |
| worker init | `syncFromPersistenceBeforeInit`, `prepareTypeDuckFilesystem`, `assertTypeDuckAssetsReady`, `TypeDuckRuntime.init` | FS sync direction and asset readiness occur before native init. |

### Concrete Playwright acceptance criteria

A Phase 10 Playwright/browser validation is acceptable only if all applicable criteria below pass or are recorded as reproducible blockers:

- The test starts the real TypeDuck-Web app through its normal dev/build server and navigates to the existing app route, expected from research to be `/web/`.
- The browser test interacts with DOM controls that a user would use: input area, candidate rows, page buttons or page keys, delete affordance, preferences/customize controls, and deploy controls.
- Selectors prefer roles/labels/text. Add `data-testid` only when the upstream UI has no stable semantic locator; added selectors must not change visual layout.
- The composition test asserts visible preedit/composition and at least one visible candidate when the deterministic fixture schema produces candidates.
- The paging test asserts a changed page number, changed candidate list, disabled previous/next state, or another app-visible paging state.
- The selection test asserts the selected candidate's text is committed exactly once to the app-visible output.
- The deletion test asserts the deleted candidate is no longer present in its prior position, a replacement candidate appears, or the app reports a deterministic unsupported-deletion blocker.
- The deploy test asserts the existing deploy status path reaches success or a deterministic failure state with command/output evidence.
- The customize test asserts a supported preference/custom patch path changes state and remains usable after deploy/sync, or records the exact unsupported option gap.
- The persistence smoke test calls or observes `syncfs(true)` before init and `syncfs(false)` after mutation through the Phase 9 helpers, then reloads/reinitializes and verifies persisted state.
- The browser console has no unhandled exceptions from `TypeDuckResponseError`, `TypeDuckLifecycleError`, missing `cwrap`, missing `UTF8ToString`, missing `FS.syncfs`, null response pointers, or unhandled worker messages.
- If Emscripten, WASM artifact, Playwright browsers, upstream clone, or browser APIs are unavailable, the test result is `BLOCKED`, includes the exact command and output, and does not count as an E2E pass.

## State And Error Contract

| State | Required behavior |
|-------|-------------------|
| Initial loading | Preserve upstream loading/initialization UI. Worker initializes one active Yune runtime only after Module readiness, persistence sync-from, asset prep, and asset readiness checks. |
| Ready | Input area is enabled/focusable and existing TypeDuck-Web UI is visible. No new Yune onboarding or redesign shell appears. |
| Composing | Preedit/composition, caret/highlight, candidate panel, page state, and candidate comments are visible where upstream renders them. |
| No composition | Candidate panel is hidden or upstream empty state is shown; empty state copy remains `No composition` only if new test-accessible copy is required. |
| Candidate highlighted | Highlight/focus is visible through upstream styling and available through DOM state/test selector; color alone is not the only E2E assertion. |
| Candidate selected | Commit output is appended exactly once and composition state updates according to upstream behavior. |
| Candidate deleted | Existing deletion behavior is preserved; no new confirmation prompt is introduced. |
| Deploy pending | Preserve upstream deploy status event/notification behavior. If no visual pending state exists, E2E may observe worker/action state but must not add a redesign. |
| Deploy success/failure | Existing status/notification path reports success or deterministic failure; failures include next action. |
| Customize success/failure | Existing preferences/customize path reports success or deterministic unsupported-option/adapter failure. |
| Persistence sync failed | Do not continue silently. Surface deterministic `TypeDuck filesystem sync failed`-style error and record direction: `fromPersistence` or `toPersistence`. |
| Missing assets | Do not fabricate placeholder YAML. Report missing virtual paths and do not count browser E2E as passed. |
| Unsupported key/action | Report unsupported key/action deterministically; do not map to a different key or silently no-op required flows. |
| Cleanup/reload | Cleanup is idempotent through `TypeDuckRuntime.cleanup`; operations after cleanup throw deterministic lifecycle errors rather than touching a consumed native pointer. |

## Accessibility Contract

- Preserve upstream TypeDuck-Web accessibility behavior; improve only when needed for reliable E2E locators and without visual redesign.
- The main text input must be focusable from the browser test and should be locatable by role/name or a stable test id.
- Candidate rows must expose either visible text, role/name, or stable `data-testid` values that include page-relative index.
- Page controls must have distinguishable previous/next semantics through accessible names, text, title, or stable `data-testid` values.
- Preference/customize and deploy controls must be locatable without styling-class selectors where practical.
- Candidate highlight, deploy status, errors, and persistence blockers must not rely on color alone for E2E observability.
- Preserve user text exactly, including non-ASCII candidate, preedit, comment, and commit text.
- Do not add audio-only, animation-only, hover-only, or timing-only states as required validation signals.
